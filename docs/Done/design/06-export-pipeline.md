# 06 — Export Pipeline

## The principle

Design Mode produces a `DesignGraph`. The export pipeline converts it to
the formats users want:
- **Mermaid `.mmd`** — for rendering on mermaid.ai or in documentation
- **JSON `.dgraph.json`** — for save/load (covered in doc 05)
- **C# stub source** — for pasting into a project as a starting point
- **PNG/SVG** — for embedding in documents (reuses Analyze Mode export)

All exports go through a single `DesignExporter` class with one method per
format. Each method takes a `DesignGraph` and returns the output as a string
or writes it to a file.

## Architecture

```csharp
public static class DesignExporter
{
    public static string ToMermaid(DesignGraph graph);
    public static string ToCSharpStub(DesignGraph graph);
    public static void SaveJson(DesignGraph graph, string filePath);
    public static DesignGraph LoadJson(string filePath);

    // Internal converters
    internal static TypeGraph ToTypeGraph(DesignGraph design);
    internal static List<DesignEdge> MergeAutoImplementedProperties(DesignGraph graph);
}
```

The Mermaid export delegates to the existing `MermaidGraphExporter` by
converting `DesignGraph` → `TypeGraph` first. This guarantees identical
output between Analyze Mode and Design Mode for the same content.

## Mermaid export

```csharp
public static string ToMermaid(DesignGraph graph)
{
    var typeGraph = ToTypeGraph(graph);
    return MermaidGraphExporter.BuildDiagram(typeGraph);
}
```

This reuses all the sanitization logic added in the previous plan
(global namespace handling, colon-in-title escaping, angle bracket stripping).

## C# stub source export

The C# stub is minimal but compilable. It generates:

```csharp
// File: DesignGraph.cs
// Auto-generated from MermaidDiagramExporter Design Mode
// Namespace: MyApp
namespace MyApp;

public class Animal
{
    public string Name { get; set; }
    public int Age { get; set; }

    public Animal(string name, int age)
    {
        Name = name;
        Age = age;
    }

    public string GetName() { return Name; }
}

public class Dog : Animal
{
    public string Breed { get; set; }

    public Dog(string name, int age, string breed) : base(name, age)
    {
        Breed = breed;
    }
}
```

### Generation rules

1. **Namespaces**: one `namespace {Name};` block per `DesignNamespace`.
2. **Class declarations**:
   - `public class {Name}` for `ClassKind.Class`
   - `public interface {Name}` for `ClassKind.Interface`
   - `public enum {Name}` for `ClassKind.Enum`
   - `public struct {Name}` for `ClassKind.Struct`
   - `public static class {Name}` for `ClassKind.StaticClass`
   - `public abstract class {Name}` for `ClassKind.AbstractClass`
3. **Inheritance**: `: BaseClassName` for `EdgeKind.Inheritance` edges
4. **Interfaces**: `, IInterfaceName` (appended to base list) for `EdgeKind.Implements`
5. **Fields**: `public {TypeName} {Name};`
6. **Properties**: `public {TypeName} {Name} { get; set; }` (auto-implemented)
7. **Methods**: `public {ReturnType} {Name}({params}) { }` (empty body)
8. **Constructors**: `public {Name}({params}) { }` (empty body, no field init)
9. **Visibility**: maps to `public`/`private`/`protected`/`internal`

### Special handling

- **Enums**: members become enum values (`MemberName = 0,` etc.). No
  properties/methods allowed on enums in v1.
- **Static classes**: all members get `static` modifier.
- **Abstract classes/methods**: abstract methods get `abstract` modifier,
  no body (`public abstract void Foo();`).

### Edge cases

- **Empty class**: still generates `public class Foo { }` (valid C#).
- **Reserved words as names**: not validated in v1. The user is responsible
  for using valid C# identifiers. (Validation is in doc 03's "Inline editing"
  section.)
- **Generic types**: not supported in v1. `TypeName` is treated as an opaque
  string (`List<Foo>` works as-is since it's already valid C# syntax).

### Output format

```csharp
// Auto-generated by MermaidDiagramExporter Design Mode
// Date: 2026-06-23 12:34:56 UTC
// Source: MyDesign.dgraph.json

using System;

namespace MyApp
{
    public class Animal
    {
        public string Name { get; set; }
        // ...
    }
}
```

Each namespace gets its own block. Classes within a namespace are sorted
alphabetically. Members within a class are sorted: fields first, then
properties, then constructors, then methods.

## Export validation

Before exporting, the exporter validates the `DesignGraph`:

```csharp
public static class DesignValidator
{
    public static List<string> Validate(DesignGraph graph)
    {
        var errors = new List<string>();

        // Check for duplicate class names within the same namespace
        var duplicates = graph.Classes
            .GroupBy(c => (c.Namespace, c.Name))
            .Where(g => g.Count() > 1);
        foreach (var dup in duplicates)
            errors.Add($"Duplicate class '{dup.Key.Name}' in namespace '{dup.Key.Namespace}'");

        // Check for edges referencing non-existent classes
        var classIds = graph.Classes.Select(c => c.Id).ToHashSet();
        foreach (var edge in graph.Edges)
        {
            if (!classIds.Contains(edge.FromClassId))
                errors.Add($"Edge {edge.Id} references missing source class {edge.FromClassId}");
            if (!classIds.Contains(edge.ToClassId))
                errors.Add($"Edge {edge.Id} references missing target class {edge.ToClassId}");
        }

        // Check for self-edges on inheritance/implements (invalid)
        foreach (var edge in graph.Edges.Where(e => e.Kind is EdgeKind.Inheritance or EdgeKind.Implements))
        {
            if (edge.FromClassId == edge.ToClassId)
                errors.Add($"Class cannot inherit from itself: {edge.FromClassId}");
        }

        // Check for cycles in inheritance (would be a compile error in C#)
        var cycles = FindInheritanceCycles(graph);
        foreach (var cycle in cycles)
            errors.Add($"Inheritance cycle detected: {string.Join(" -> ", cycle)}");

        return errors;
    }
}
```

If validation fails, the export shows an error dialog with the list of issues.
The user can fix them and retry, or export anyway (with a confirmation
dialog warning that the output may not compile).

## Conversion: DesignGraph → TypeGraph

```csharp
internal static TypeGraph ToTypeGraph(DesignGraph design)
{
    var nodes = design.Classes.Select(ToTypeNode).ToList();
    var edges = design.Edges.Select(ToTypeEdge).ToList();
    var groups = BuildNamespaceGroups(design);

    return new TypeGraph(
        title: design.Title,
        nodes: nodes,
        edges: edges,
        groups: groups,
        metadata: new TypeGraphMetadata
        {
            GeneratedAtUtc = DateTime.UtcNow,
            SourceKind = GraphSourceKind.Design,
            Options = new GraphBuildOptions(),
            SourceDescription = $"Design: {design.Title}"
        }
    );
}

private static TypeNodeData ToTypeNode(DesignClass cls)
{
    return new TypeNodeData
    {
        Id = cls.Id,
        DisplayName = cls.Name,
        Namespace = cls.Namespace,
        Kind = ToTypeNodeKind(cls.Kind),
        Members = cls.Members.Select(ToTypeMember).ToList(),
        Stereotype = cls.Stereotype
    };
}

private static TypeEdgeData ToTypeEdge(DesignEdge edge)
{
    return new TypeEdgeData
    {
        FromNodeId = edge.FromClassId,
        ToNodeId = edge.ToClassId,
        Kind = ToTypeEdgeKind(edge.Kind, edge.Label),
        Label = edge.Label ?? ""
    };
}
```

### `DesignEdgeKind` → `TypeEdgeKind` mapping

`TypeEdgeKind` only has three values (`Inheritance`, `Implements`,
`Association`), but `DesignEdgeKind` has six
(`Association`, `Inheritance`, `Implements`, `Dependency`, `Aggregation`,
`Composition`). The converter must map the extra three deliberately, not
silently:

| `DesignEdgeKind` | `TypeEdgeKind` | Notes |
|------------------|----------------|-------|
| `Inheritance` | `Inheritance` | direct mapping |
| `Implements` | `Implements` | direct mapping |
| `Association` | `Association` | direct mapping |
| `Dependency` | `Association` | Mermaid has no separate "dependency" edge kind; falls back to association with the label "depends on" if no custom label is set |
| `Aggregation` | `Association` | Mermaid has no separate "aggregation" edge kind; falls back to association with the label "has" if no custom label is set |
| `Composition` | `Association` | Mermaid has no separate "composition" edge kind; falls back to association with the label "owns" if no custom label is set |

```csharp
private static TypeEdgeKind ToTypeEdgeKind(EdgeKind kind, string? label)
{
    return kind switch
    {
        EdgeKind.Inheritance => TypeEdgeKind.Inheritance,
        EdgeKind.Implements => TypeEdgeKind.Implements,
        EdgeKind.Association => TypeEdgeKind.Association,
        // The following three all collapse to Association in Mermaid.
        // The label preserves the semantic distinction in the output.
        EdgeKind.Dependency => TypeEdgeKind.Association,
        EdgeKind.Aggregation => TypeEdgeKind.Association,
        EdgeKind.Composition => TypeEdgeKind.Association,
        _ => TypeEdgeKind.Association
    };
}
```

This is a **lossy conversion** — the `DesignEdgeKind` distinction is lost in
the Mermaid export. The validator should warn users about this:

```csharp
// In DesignValidator
foreach (var edge in graph.Edges.Where(e => e.Kind is EdgeKind.Dependency
    or EdgeKind.Aggregation or EdgeKind.Composition))
{
    warnings.Add($"Edge {edge.Id} uses {edge.Kind} which Mermaid renders as Association. " +
                 "The semantic distinction is preserved in the label but not in the diagram type.");
}
```

The C# stub export keeps all six `DesignEdgeKind` values (C# has no
association/aggregation/composition distinction in syntax — they're all
just fields or properties).

The `Id` field uses the `DesignClass.Id` GUID directly, so the layout engine
and rendering pipeline work without translation.

## Risks

- **C# stub compilation**: the generated stub must compile in a fresh .NET
  project. Validation: parse with Roslyn (`CSharpSyntaxTree.ParseText`) and
  check for syntax errors. If parsing fails, the exporter has a bug.
- **Namespace nesting depth**: deeply nested namespaces produce deeply nested
  C# code, which is hard to read. Limit to 3 levels in v1.
- **Mermaid syntax limits**: Mermaid class diagrams have limits on member
  count per class (practical limit ~30, hard limit unknown). The exporter
  should warn if a class exceeds 30 members.
- **Round-trip drift**: a `DesignGraph` → `TypeGraph` → `DesignGraph` round-trip
  might not produce an identical graph (e.g. auto-implemented properties
  get expanded into fields + getters/setters). Acceptable for v1; document
  the lossy transformations.
